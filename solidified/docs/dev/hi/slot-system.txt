# स्लॉट और विजेट सिस्टम

मॉड्यूल UI विजेट को नामित **स्लॉट** में इंजेक्ट करते हैं — ये पूर्व-निर्धारित लेआउट क्षेत्र हैं जिन्हें शेल (`Layout.tsx`) रेंडर करता है। इससे फ़ीचर कोड लेआउट से अलग रहता है।

हर विजेट एक **स्थिर id** के साथ केंद्रीय विजेट रजिस्ट्री (`module-registry.ts`) में रजिस्टर होता है। ये id यूज़र-कॉन्फ़िगरेबल विजेट लेआउट की पहचान इकाई हैं, इसलिए एक बार शिप होने के बाद इन्हें कभी न बदलें।

## उपलब्ध स्लॉट

| स्लॉट की | कहाँ रेंडर होता है |
|----------|------------------|
| `right` | राइट साइडबार (डेस्कटॉप) |
| `leftBottom` | लेफ्ट साइडबार का निचला हिस्सा |
| `header` | मुख्य कंटेंट एरिया का सबसे ऊपरी हिस्सा, `mainTop` से ऊपर — हमेशा पूरी चौड़ाई में, सिंगल कॉलम |
| `mainTop` | मुख्य कंटेंट एरिया का ऊपरी हिस्सा — CSS कॉलम के ज़रिए मेसनरी-पैक्ड, ताकि अलग-अलग ऊँचाई वाले विजेट बिना खाली जगह छोड़े फिट हो जाएँ |
| `footer` | मुख्य कंटेंट एरिया का सबसे निचला हिस्सा, पेज कंटेंट के नीचे — हमेशा पूरी चौड़ाई में, सिंगल कॉलम, `header` जैसा |
| `rightVisitor` | विज़िटर को दिखने वाला राइट साइडबार |

## मॉड्यूल में विजेट घोषित करना

```typescript
registerModule({
  id: "network",
  // …
  widgets: [
    {
      id: "network.filters",            // स्थिर, सहेजा जाता है — कभी न बदलें
      label: "Stream Filters",          // विजेट पिकर UI में दिखता है
      loader: () => import("./widgets/StreamFiltersWidget"),
      slot: "right",
    },
  ],
});
```

`WidgetDef` फ़ील्ड (देखें `shared/types/module.types.ts`):

- `id` — स्थिर पहचानकर्ता, कन्वेंशन `"<moduleId>.<name>"`।
- `label` — मानव-पठनीय नाम (string या i18n के लिए `() => string`)।
- `loader` — `() => Promise<{ default: Component }>`।
- `slot` — विजेट किस लेआउट क्षेत्र में रेंडर होगा।
- `defaultModules` — वे मॉड्यूल id जहाँ विजेट डिफ़ॉल्ट रूप से दिखता है। डिफ़ॉल्ट: रजिस्टर करने वाला मॉड्यूल। उदाहरण: channel विजेट `["channel", "profile"]` घोषित करते हैं ताकि वे प्रोफ़ाइल पेज पर भी दिखें।
- `contexts` — वे मॉड्यूल id जहाँ यूज़र विजेट रख सकता है, या `"any"`। डिफ़ॉल्ट: `defaultModules`।
- `global` — हर पेज पर हमेशा माउंटेड, नेविगेशन पर कभी टियर-डाउन नहीं।
- `visitorVisible` — `false` का अर्थ है केवल प्रमाणीकृत लोकल यूज़र ही विजेट देखें; इसे उन विजेट पर लगाएँ जो व्यूअर-निजी डेटा दिखाते हैं (ड्राफ्ट, बुकमार्क) ताकि सार्वजनिक पेजों के विज़िटर उन्हें कभी माउंट न करें। डिफ़ॉल्ट `true`।

## ग्लोबल बनाम मॉड्यूल-लोकल विजेट

डिफ़ॉल्ट रूप से विजेट **मॉड्यूल-लोकल** होते हैं: केवल तब रेंडर होते हैं जब `defaultModules` में सूचीबद्ध कोई मॉड्यूल एक्टिव हो, और नेविगेट करते ही अनमाउंट हो जाते हैं।

**ग्लोबल विजेट** (`global: true`) पूरे सेशन में माउंटेड रहते हैं — नेविगेशन के बाद भी। साझा विजेट `shared/slots.ts` में हैं:

```typescript
// shared/slots.ts
export const notificationsWidget: WidgetDef = {
  id: "shared.notifications",
  label: "Notifications",
  loader: () => import("./widgets/notifications/NotificationsAside"),
  slot: "right",
  contexts: "any",
  global: true,
};
```

एक विजेट id केवल एक बार रजिस्टर हो सकता है — डुप्लिकेट रजिस्ट्रेशन कंसोल चेतावनी के साथ अनदेखा कर दिए जाते हैं।

## यूज़र लेआउट

यूज़र यह बदल सकते हैं कि किस पेज पर कौन से विजेट दिखें। लेआउट pconfig (cat `spa`, key `widget_layout`) में JSON के रूप में रहता है:

```json
{
  "version": 1,
  "modules": {
    "hq":      { "right": ["shared.notifications", "network.savedSearch"] },
    "channel": { "right": ["channel.archive", "channel.connections"] }
  }
}
```

- मॉड्यूल/स्लॉट एंट्री की अनुपस्थिति → रजिस्ट्री डिफ़ॉल्ट लागू होते हैं।
- स्पष्ट खाली array → यूज़र ने उस स्लॉट से सभी विजेट हटा दिए।
- `POST /api/widget-layout` से सहेजा जाता है (`{ layout: … }`, `{ layout: null }` सब कुछ रीसेट करता है); बूट पर `GET /api/pconfig` से लोड होता है।

क्लाइंट स्टेट `src/shared/store/widget-layout.ts` में है: `layoutFor(moduleId, slot)`, `saveSlotLayout(moduleId, slot, ids | null)`, `saveWidgetLayout(layout | null)`। सेव ऑप्टिमिस्टिक हैं (विफलता पर रोलबैक); localStorage (`hz-widget-layout`) लेआउट कैश करता है ताकि pconfig लोड होते समय साइडबार डिफ़ॉल्ट न दिखाए। लोकल यूज़र के लिए सर्वर का मान — उसकी अनुपस्थिति सहित — हमेशा जीतता है।

सहेजे गए id जो अज्ञात, ग्लोबल, संदर्भ-निषिद्ध, या अनइंस्टॉल-ऐप विजेट की ओर इशारा करते हैं, रेंडर के समय चुपचाप हटा दिए जाते हैं: लेआउट कोड परिवर्तनों से अधिक समय तक जीवित रहते हैं।

### विज़िटर को स्वामी की व्यवस्था दिखती है

`widget_layout` उन प्रति-चैनल प्राथमिकताओं में शामिल है जो `GET /api/pconfig?channel=<nick>` देता है (`Pconfig.php` में `channelSpa()`) — वही पेलोड जो चैनल की थीम ले जाता है। `useChannelTheme` इसे एक अलग `pageLayout` सिग्नल में भेजता है (`initPageWidgetLayout` / `pageLayoutFor`), जो चैनल पेज छोड़ने पर साफ़ हो जाता है। `Slot` व्यूअर की भूमिका से स्रोत चुनता है: अपने पेजों पर (`viewerRole() === "owner"`) आपका लेआउट लागू होता है; किसी और के चैनल पेजों पर — लॉग-इन लोकल यूज़र सहित — पेज स्वामी का लेआउट लागू होता है। पेज-स्वामी लेआउट विज़िटर के डिवाइस पर कभी सहेजा नहीं जाता।

## संपादन मोड

लोकल यूज़र को राइट-साइडबार हेडर में एक पेंसिल टॉगल मिलता है (सभी ब्रेकपॉइंट पर दिखता है — मोबाइल पर साइडबार ड्रॉअर है, `xl` पर कॉलम)। संपादन के दौरान हर विजेट में ऊपर/नीचे/हटाएँ नियंत्रण वाला हेडर और एक निष्क्रिय, धुँधला पूर्वावलोकन दिखता है; सूची के नीचे "विजेट जोड़ें" पिकर (वर्तमान मॉड्यूल में अनुमत विजेट जिनका ऐप इन्स्टॉल है) और "डिफ़ॉल्ट पर रीसेट करें" बटन (केवल तब जब पेज कस्टमाइज़ हो) होते हैं। हर क्रिया `saveSlotLayout` से तुरंत और ऑप्टिमिस्टिक रूप से सहेजी जाती है; विफलताएँ रोलबैक होती हैं और एरर टोस्ट दिखता है।

स्टेट: `widget-layout.ts` में `editingWidgets` / `setEditingWidgets`; केवल `editable` prop के साथ रेंडर हुए `<Slot>` संपादन UI दिखाते हैं। ग्लोबल विजेट संपादन-योग्य नहीं हैं। पेंसिल और संपादन UI केवल आपके अपने पेजों पर दिखते हैं (`viewerRole() === "owner"`) — संपादन हमेशा आपके अपने लेआउट पर लागू होता है, और किसी और के पेज पर जाते ही संपादन मोड अपने आप बंद हो जाता है।

## Slot कॉम्पोनेंट

`src/shared/views/Slot.tsx` नाम से एक स्लॉट रेंडर करता है:

```tsx
<Slot name="right" />
<Slot name="right" moduleId="network" />  // स्पष्ट मॉड्यूल ओवरराइड
```

अगर `moduleId` न दिया जाए तो एक्टिव मॉड्यूल मौजूदा URL पाथ से अनुमान लगाया जाता है।

`Slot` अंदर से रेंडर करता है:

1. स्लॉट के सभी **ग्लोबल** विजेट (हमेशा माउंटेड, कभी टियर-डाउन नहीं)।
2. **लोकल** विजेट: एक्टिव मॉड्यूल के लिए यूज़र का सहेजा लेआउट अगर मौजूद हो, वरना वे विजेट जिनके `defaultModules` में एक्टिव मॉड्यूल शामिल है (नेविगेशन पर स्वैप होते हैं)।

लोकल विजेट तब स्किप हो जाते हैं जब एक्टिव मॉड्यूल का `appName` ऐप इन्स्टॉल न हो; हर विजेट अपने *स्वामी* मॉड्यूल के ऐप से भी गेट होता है, और `visitorVisible: false` विजेट गैर-लोकल व्यूअर के लिए स्किप हो जाते हैं (यह फ़िल्टर ग्लोबल विजेट पर भी लागू होता है)।

## रजिस्ट्री लुकअप

- `getWidget(id)` — एक रजिस्टर्ड विजेट, या `null`।
- `getAllWidgets()` — रजिस्ट्रेशन क्रम में सभी रजिस्टर्ड विजेट (पिकर UI, लेआउट सत्यापन)।
- `resolveGlobalSlots(slot)` / `resolveModuleSlot(slot, moduleId)` — वे `RegisteredWidget` सूचियाँ जो `Slot` डिफ़ॉल्ट रूप से रेंडर करता है।
- `widgetAllowedIn(widget, moduleId)` — क्या यूज़र विजेट को उस मॉड्यूल के पेजों पर रख सकता है (`contexts` जाँच)।

## लेज़ी कैशिंग

`module-registry.ts` में `getLazy(loader)` हर लोडर को ठीक एक बार `lazy()` में रैप करता है, ताकि रिएक्टिव मेमो के रि-कम्प्यूट होने पर Solid कॉम्पोनेंट को फिर से माउंट न करे:

```typescript
const lazyCache = new WeakMap<SlotLoader, Component>();
export function getLazy(loader: SlotLoader): Component {
  if (!lazyCache.has(loader)) lazyCache.set(loader, lazy(loader));
  return lazyCache.get(loader)!;
}
```

## रिएक्टिविटी: widgetVersion

`widgetVersion` एक सिग्नल है जो हर बार किसी मॉड्यूल के विजेट रजिस्टर होने पर बढ़ता है। `Slot` कॉम्पोनेंट इसे ट्रैक करता है ताकि async मॉड्यूल इम्पोर्ट के बाद रजिस्टर हुए विजेट भी तुरंत दिखें:

```typescript
const widgetVersion = getWidgetVersion();
const globalWidgets = createMemo(() => {
  widgetVersion(); // सब्सक्राइब करें
  return resolveGlobalSlots(props.name).map(getLazy);
});
```

## पुराना तरीक़ा: `slots`

मॉड्यूल का पुराना `slots: { right: [loader] }` फ़ील्ड अब पदावनत (deprecated) है और रजिस्ट्री इसे **अनदेखा** करती है (गैर-खाली होने पर कंसोल चेतावनी आती है)। `widgets` पर माइग्रेट करें।
